# Init CLI — Scaffold Joysmith Harness — Atomic Spec

> **Parent Brief:** `docs/briefs/2026-03-23-joysmith-cli-plugin.md`
> **Status:** Ready
> **Date:** 2026-03-23
> **Estimated scope:** 2-3 sessions / 8-10 files / ~500 lines

---

## What

Build `npx joysmith init` — a CLI command that scaffolds the complete Joysmith harness into any project. It auto-detects the tech stack, creates or improves CLAUDE.md with behavioral boundaries and correct commands, sets up the docs/ directory structure, copies all workflow skills into .claude/skills/, and optionally registers hooks.

## Why

This is the main distribution mechanism. Without it, users have to manually copy templates from the Joysmith repo — which nobody will do. `npx joysmith init` makes adoption a one-command experience.

## Acceptance Criteria

- [ ] `npx joysmith init` runs successfully in a project directory
- [ ] Creates docs/ directory structure: briefs/, specs/, discoveries/, contracts/, decisions/
- [ ] Copies workflow skills to .claude/skills/: joysmith.md, new-feature.md, decompose.md, session-end.md
- [ ] Copies templates to docs/templates/: ATOMIC_SPEC_TEMPLATE.md, FEATURE_BRIEF_TEMPLATE.md, IMPLEMENTATION_PLAN_TEMPLATE.md, BOUNDARY_FRAMEWORK.md
- [ ] If no CLAUDE.md exists: creates one using the template, populated with detected stack commands
- [ ] If CLAUDE.md exists: improves it — adds missing sections (boundaries, architecture, workflow) without destroying existing content
- [ ] Detects tech stack using `detectStack()` (Spec 8) and populates build/test/lint commands
- [ ] Prints a summary of what was created/modified
- [ ] Prints next steps: "Run Claude Code in this project and try `/joysmith` to assess your harness"
- [ ] Does NOT overwrite files that already exist (skills, templates) — warns and skips
- [ ] Has a `--force` flag to overwrite existing files when explicitly requested
- [ ] Works with npx (no global install required)
- [ ] Build passes
- [ ] Tests pass

## Constraints

- MUST be an npm package publishable to npm registry as `joysmith`
- MUST work with `npx` (zero-install execution)
- MUST NOT require Node.js-specific project setup — the CLI runs via npx but the target project can be any language
- MUST NOT delete or overwrite user files without `--force`
- MUST handle existing partial harness (some dirs exist, some don't) gracefully
- SHOULD be a single `init` command with no subcommands (keep it simple for v1)
- SHOULD complete in under 5 seconds (it's just copying files)

## Approach

The CLI uses a minimal argument parser (commander or similar — one dependency). The `init` command:

1. Calls `detectStack()` to identify the project
2. Creates docs/ subdirectories (mkdir -p, skip if exists)
3. Copies template files to docs/templates/ (skip if exists, unless --force)
4. Copies skill files to .claude/skills/ (skip if exists, unless --force)
5. If no CLAUDE.md: generates one from the template with detected commands filled in
6. If CLAUDE.md exists: reads it, identifies missing sections, appends/merges them
7. Prints summary and next steps

The CLAUDE.md improvement logic (step 6) is the trickiest part. Strategy: parse the existing file into sections (by `## ` headers), identify which Joysmith sections are missing (Behavioral Boundaries, Architecture, Development Workflow, Common Gotchas), and append them. Never modify existing sections.

**Rejected alternative:** Interactive prompts (inquirer-style) asking about tech stack, team size, etc. Auto-detection is faster and more reliable. Users can customize after init.

## Edge Cases

| Scenario | Expected Behavior |
|----------|------------------|
| Empty directory (no files at all) | Create CLAUDE.md + dirs + skills, detect as "unknown" stack |
| .claude/skills/ already has user skills | Don't touch existing skills, add Joysmith skills alongside |
| CLAUDE.md has custom boundaries | Preserve them, don't add duplicate boundary section |
| Read-only filesystem | Error with clear message: "Cannot write to project directory" |
| Running init twice | Second run is a no-op (skips existing files), prints "already initialized" |
| Monorepo root | Scaffold at root level, note in output that components may want their own CLAUDE.md |
| No package.json in target (it's a Go project) | Still works — CLI runs via npx, target detection finds go.mod |
